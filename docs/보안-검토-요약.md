# 보안 검토 요약 (재검토)

## 1. 관리자 코드

| 항목 | 상태 |
|------|------|
| 서버에서만 사용 | ✅ `verify-code`는 `ADMIN_CODE`만 참조. `NEXT_PUBLIC_ADMIN_CODE` 미사용으로 클라이언트 번들에 코드 값 미포함 |
| 입력란 마스킹 | ✅ 메인 가입·프로필 완성 페이지에서 슈퍼관리자 코드 입력이 `type="password"`, `autoComplete="one-time-code"` |
| check-env | ✅ 관리자 코드 항목은 `ADMIN_CODE`만 표시 |

**권장**: `.env.local`에 `NEXT_PUBLIC_ADMIN_CODE`가 있다면 제거하고 `ADMIN_CODE=원하는값`만 사용.

---

## 2. API 인증

### 관리자 API (Bearer + admin 역할 + 본인 학교)

- `count-by-school`, `teachers`, `teacher-summaries`, `mileage-by-email`, `points-by-email`, `plan-by-email`, `result-report-by-email`, `diagnosis-upload`, `diagnosis-settings`, `diagnosis-title`, `verify-teacher-email`, `reset-password`, `reset-all-data`  
→ 모두 토큰 검증 후 `role === 'admin'` 및 요청 `schoolName`이 호출자 학교와 일치할 때만 처리.

### AI API (Bearer + 로그인 사용자)

- `ai-recommend` — ✅ 기존 적용
- `ai-classify-mileage` — ✅ Bearer + `getUser(token)` 추가, 대시보드 마일리지 호출 시 토큰 전달
- `ai-summarize-reflections` — ✅ Bearer + `getUser(token)` 추가, 성찰 페이지 호출 시 토큰 전달
- `ai-refine-mileage` — ✅ Bearer + `getUser(token)` 추가 (호출처 없어도 미인증 호출 방지)

### 기타 인증 필요 API

- `auth/complete-profile` — Bearer 필수, 관리자 시 학교당 3명 제한 서버 검증
- `points/me`, `points/init`, `points/login`, `points/school-settings` — Bearer 또는 토큰 검증
- `diagnosis-settings`, `diagnosis-questions`, `school-category-settings`, `school-diagnosis-settings`, `mileage-relative-difficulty` — Bearer 검증
- `account/delete`, `account/reset-data` — Bearer 검증

### 인증 없는 API (의도된 공개)

- `verify-code` — 관리자 코드 문자열만 검증; 코드 자체는 서버 전용(`ADMIN_CODE`)이라 클라이언트에 노출되지 않음.

## 3. 환경 변수

- **서버 전용**(클라이언트 노출 금지): `SUPABASE_SERVICE_ROLE_KEY`, `ADMIN_CODE`, `GOOGLE_CLOUD_SERVICE_ACCOUNT_JSON`(Vertex AI 서비스 계정)  
  → 코드베이스 기준으로 `NEXT_PUBLIC_` 접두사 없이 사용 중.
- **클라이언트 노출 가능**: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`  
  → 공개 가능한 값만 사용.

---

## 4. RLS

- **적용 대상**: `reflection_drafts`, `user_preferences`, `mileage_entries`  
  → `auth.jwt() ->> 'email' = user_email` 조건으로 본인만 접근.
- **적용 방법**: `supabase/rls-security-fix.sql`을 Supabase 대시보드 SQL Editor에서 수동 실행 필요 (앱에서 자동 실행 안 함).
- **미적용 테이블**: `diagnosis_results`, `development_plans`, `user_points` 등 — 앱에서는 `user_email`로만 조회하지만, RLS 미적용 시 anon key 악용 이론적 가능성 있음. 필요 시 동일 방식으로 정책 추가 권장.

---

## 5. 학교당 관리자 3명 제한

- **프론트**: 메인 가입·프로필 완성 시 `verify-code`에 `schoolName` 전달 후 `adminCount >= 3`이면 가입/완성 차단.
- **서버**: `complete-profile`에서 `role === 'admin'`일 때 해당 학교 관리자 수를 세어 3명 이상이면 403 반환.

---

## 6. 이번 검토에서 수정·보완한 사항

1. **AI API 인증 보강**  
   `ai-classify-mileage`, `ai-summarize-reflections`, `ai-refine-mileage`에 Bearer 검증 및 `getUser(token)` 추가, 호출처에서 `Authorization: Bearer ${token}` 전달하도록 수정.
2. **보안 현황 문서 반영**  
   `docs/보안-현황.md`에 위 AI API 3개와 관리자 코드(서버 전용·입력 마스킹) 내용 반영.

---

## 7. 요약

- **관리자 코드**: 서버 전용 변수만 사용, 입력란 마스킹 적용됨.
- **민감 API**: 관리자·AI·포인트·계정 관련 API는 Bearer 및 역할/학교 검증 적용됨.
- **RLS**: 3개 테이블 정책은 스크립트 실행 시 적용; 나머지 테이블은 선택적으로 동일 방식 확장 가능.
- **추가 권장**: 다른 테이블 RLS, 레이트 리밋, 관리자 코드 주기적 변경 등은 `docs/보안-현황.md` 참고.
